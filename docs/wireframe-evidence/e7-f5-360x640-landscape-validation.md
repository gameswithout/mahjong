# E7.F5 — §9.2 simultaneous-visibility wireframe validation

**Status: AI-assisted validation complete. Human UX Lead sign-off not performed — see "What this is not" below.**

## What was built

- `client/MatchTable.tsx` + `client/match-table.css` — a static layout
  prototype of the §9.2 match table, driven entirely by mock data
  (`client/matchTableMockData.ts`). No server wiring, no real input
  handling: this proves the *layout*, not the production match view (E8).
- `wireframe.html` / `client/wireframe-main.tsx` — a second Vite entry
  point serving the prototype standalone, with a scenario toggle between a
  normal turn and an urgent-countdown / Pass-only claim window.
- `scripts/validate-match-table-wireframe.mjs` — a Playwright-driven check
  that loads the page in a real headless Chromium at exactly 640×360,
  measures every required element's presence and bounds, and fails the
  process (non-zero exit) if anything is missing, clipped outside the
  table, or under its accessible touch-target minimum. Re-run it after any
  future layout change: `npm run dev` in one terminal, then
  `node scripts/validate-match-table-wireframe.mjs`.

## Why 640×360, not 360×640

§9.2 states the gate is "the 360 by 640 CSS pixel landscape minimum" — a
360×640 phone *rotated into landscape*, i.e. an effective viewport of
**640 wide × 360 tall**. That's the number tested throughout.

## Result

`node scripts/validate-match-table-wireframe.mjs` exits 0. Full measured
report:

- Match table renders at exactly 640×360 with **zero overflow** (top-left
  at `(0,0)`, bottom-right at `(640,360)`).
- Every element §9.2 requires simultaneously visible is present and
  entirely within the table's bounds (not clipped): tile identity (own
  hand, 17/17 tiles), claim source, most recent discard (pulsing outline),
  active player, dealer, all four seat wind labels (E/S/W/N), continuation
  count, countdown, and all four legal-action buttons in the mock scenario.
- Drawable-tile count (excluding the 16-tile reserve) is visible in the
  center panel.
- Every opponent's separate discard grid and exposed-meld area render
  without clipping at the mock's discard counts (3/6/5/4 across the four
  seats).
- §9.9 touch-target minimums, measured (not assumed): all 17 local hand
  tiles hit exactly 32×44 CSS px; all four action buttons are ≥44×44 CSS px
  (44–58.5 px wide, 44 px tall).
- WCAG 2.2 AA contrast, computed for every text/background pair in the
  stylesheet (relative-luminance formula, not eyeballed): all ≥4.5:1 except
  the dealer badge, which is 9px text needing the full 4.5:1 (not the
  3:1 large-text allowance) — that pair was fixed from 3.96:1 to 7.09:1 by
  darkening the badge red (`#d9534f` → `#a52a27`).
- `prefers-reduced-motion: reduce` disables the pulsing animations
  (active-player pulse, recent-discard pulse).

Evidence screenshots (regenerated by the script, captured in this pass):
`normal-turn.png`, `urgent-claim-window.png` in this directory.

## Bugs the validation caught (worth recording, since they'd recur)

1. **CSS grid/flex blowout.** Grid/flex children default to
   `min-width/min-height: auto`, which uses the content's intrinsic size
   and ignores the track size. North's 17-tile-back row forced its whole
   row wider than its grid track, overflowing the table. Fixed by setting
   `min-height: 0; min-width: 0` on every nested grid/flex level, not just
   the outermost container.
2. **`box-sizing` gap.** `.match-table` itself wasn't covered by the
   `.match-table *` border-box rule (only its *descendants* were), so its
   3px padding added on top of `width: 100%`, overflowing by 6px. Neither
   of these was visible from reading the CSS alone — both were only found
   by measuring actual rendered bounding boxes in a real browser.
3. **Missing `body { margin: 0 }`** on the standalone wireframe page (the
   main app gets this from `client/styles.css`, which the wireframe didn't
   import) — the default 8px UA-stylesheet body margin pushed everything
   off by 8px.
4. **Dealer badge contrast**, caught by computing actual ratios rather
   than eyeballing the color choice (see above).

None of these would have been caught by CSS-on-paper review; all four were
found by measuring a real rendered page.

## What this is not

- **Not a UX Lead sign-off.** §9.2 names sign-off as a UX Lead responsibility,
  and there is no UX Lead in this session to record one. Per the D6
  precedent already accepted for M1 rules certification (Product Owner
  holds Rules Lead role; certification = AI-assisted cross-checking), this
  document is the AI-assisted validation a human reviewer needs — measured
  evidence against every §9.2/§9.9 requirement in scope — not a substitute
  for that reviewer's actual sign-off. Treat this as "ready for review,"
  not "gate closed."
- **Not full responsive/device-lab coverage.** Only the mandatory 640×360
  minimum was tested. "Every supported match viewport" (larger phones,
  tablet, desktop, real iOS/Android devices) is E8.F1's device-lab pass.
- **Not full accessibility coverage.** Verified: multi-modal tile identity
  (symbol + numeral + accessible name via `aria-label`/`title`, not color
  alone), computed contrast ratios, Reduced Motion, touch-target minimums.
  Not exercised here: full keyboard navigation (arrow-key hand traversal,
  §9.3), screen-reader live-announcement behavior, High Contrast tile skin,
  200% text-scale reflow. Those need the real interactive implementation
  (E7.F3/E8), not a static layout prototype.
- **Not exhaustive discard-pile scaling.** The mock scenario uses a
  realistic mid-hand discard count (3–6 per seat). A very long hand
  (many more discards per seat) is a known scaling risk for the compact
  discard-grid area, flagged here rather than solved — E8 hardening should
  explicitly test a worst-case discard count.
- **Not the production match table.** `MatchTable.tsx` is a layout
  prototype with mock data and no interaction logic. E8 will need to wire
  it to live server state, add real input handling, and re-verify these
  same measurements hold once real data (which can be far more varied than
  one mock snapshot) is flowing through it.
