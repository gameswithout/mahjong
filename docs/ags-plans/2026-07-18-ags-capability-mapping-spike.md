# AGS Capability-Mapping Spike (dev-plan §1.3 open item 1)

- Date: 2026-07-18
- Status: First pass complete for all 12 checklist lines in
  [`mahjong-development-plan.md`](../mahjong-development-plan.md) §1.3 item 1.
  Every line below has a written go/fallback verdict. Several are marked
  **CONDITIONAL** pending a docs/behavioral check that CLI schema discovery
  alone cannot answer (noted per line) — those are follow-ups, not blockers
  to continuing design work.
- Scope: this is read-only capability discovery. No AGS namespace
  configuration, IAM client, or resource was created, updated, or deleted.
- Namespace: `gameswithout-mahjong`, profile `mahjong` (client-credentials,
  tooling host `https://gameswithout.prod.gamingservices.accelbyte.io`).

## Method

- `ags describe <service> [resource] [method]` — machine-readable, no-auth
  API surface introspection (path templates, request/response fields,
  enums). Used for every AGS-native service line below. This proves API
  *shape* (what fields/operations exist) but not runtime *behavior*
  (e.g. whether a field is actually enforced server-side) — flagged where
  that distinction matters.
- `gh api repos/AccelByte/<repo>/readme` — for the three Extend Apps
  Directory candidates (`extend-core-matchmaker`, `extend-rank-suite`,
  `extend-mmr-calculator`). These are GitHub template repos, not live AGS
  API surface, so CLI discovery doesn't apply to them.
- Prior live evidence already on file for IAM guest bootstrap, Lobby,
  Session, and Matchmaking ticket lifecycle (linked per line below) is
  reused rather than re-verified.

## Verdict table

| # | Capability (spec §) | AGS mechanism found | Verdict | Notes |
|---|---|---|---|---|
| 1 | Claim/hidden-information boundaries | Custom Mahjong match actor (not an AGS service) | **GO — already built** | Per-seat redacted views, private claim collection, and the hidden-info test suite exist in `rulesengine`/`server/match` and `mahjong-match-service`. Not an AGS capability question. |
| 2 | Token lifetimes/revocation (§15.10) | IAM `clients.update`: `oauthAccessTokenExpiration` (max 86,400 s / 24 h), `oauthRefreshTokenExpiration` (max 2,592,000 s / **30 days**); `oauth2.revoke-token`, `revoke-user`, `list-revocations` | **CONDITIONAL** | Max configurable refresh-token lifetime is a **hard 30-day ceiling** — there is no separate "90-day absolute cap, 30-day inactivity" split exposed in the client-config schema. If AGS's refresh expiry is inactivity-based (renews on use, up to the 30-day ceiling), that alone satisfies the spec's 30-day inactivity rule but not the 90-day absolute rule, which would need a thin Go wrapper checking an embedded issued-at claim. Follow-up: confirm with AccelByte docs/support whether refresh-token expiry is absolute or sliding — CLI schema doesn't say. `revoke-user`/`list-revocations` exist, consistent with family/logout-all revocation, but reuse-detection semantics need the same doc follow-up. |
| 3 | Legal-service age-gate fit (§10.3) | `legal.agreements`/`legal.policies` (ToS/Privacy versioning, per-country) + separate `iam.age-restrictions` (per-namespace/country minimum-age gate) + `iam.users.create` body field `dateOfBirth` (full `YYYY-MM-DD`) or boolean `reachMinimumAge` self-attestation | **FALLBACK triggered** | Two concrete gaps: (a) IAM's only age-collection fields are a **full DOB** or a bare **self-attestation boolean** — there is no native month/year-only field, so satisfying "no full-DOB retention" means using `reachMinimumAge` only, which loses the country-specific minimum-age enforcement `age-restrictions` is built around. (b) The primary V1 onboarding path is **guest/Device ID** (`oauth2.generate-token-for-headless-account`), which doesn't go through `users.create` at all — no AGS age-gate touches it natively. **Conclusion: the spec's month/year age-gate must be a first-party client-side gate (age-restriction store, `consent_record` fallback already scoped in §2.2a) run before Device ID login and before any `users.create` upgrade; AGS `legal` handles versioned ToS/Privacy acceptance well (accept/bulk-accept/list-accepted/CSV export all exist) and stays as primary for that half.** |
| 4 | Extend match-runtime hosting (§15.5, §15.7) | Extend Service Extension (`mahjong-match-service`) | **GO — already built** | See [match-runtime-command-flow](./2026-07-18-match-runtime-command-flow.md) and [production-seat-durability-spike](./2026-07-18-production-seat-durability-spike.md). Live AGS smoke passed; production seat/durability plan is written and awaiting approval, separate from this spike. |
| 5 | `extend-core-matchmaker` fork feasibility (§8.5) | `github.com/AccelByte/extend-core-matchmaker` — Go Extend **Override** template, gRPC server AGS calls instead of its default matchmaking function | **CONDITIONAL GO** | It is genuinely forkable (that's its entire purpose) and comes with auth/observability scaffolding pre-built. But the shipped example only matches 2 players — **none** of §8.5's actual logic (latency banding, rating band + 20s expansion, recent-opponent/block avoidance) is in the template; it all has to be authored inside the custom function. Treat the fork as "reuse the harness, write the algorithm," not "reuse the algorithm." |
| 6 | Wallet hold/escrow + idempotency (§7) | `platform.wallets`: `credit`, `debit-currency`, `pay`, `bulk-credit`, `bulk-debit`, `get-balance`. No `hold`/`reserve`/`escrow` resource or method exists anywhere under `platform.wallets`. No idempotency-key body field on `credit`/`debit-currency`/`pay` (header-based dedup, if any, isn't visible via `describe`). | **FALLBACK confirmed — decides the D13 spike question** | AGS Wallet is credit/debit/pay only; it has no reservation primitive. This settles the "wallet-as-authority vs journal-authoritative-with-mirror" question in the dev plan: **journal-authoritative is required.** The `jade-settlement` Extend Service Extension must own reserves/holds/idempotency in its own durable store, and treat Wallet purely as the execution + portal-visible mirror (credit/debit calls issued only after the first-party journal has committed). No design change needed beyond confirming this is mandatory, not optional. |
| 7 | Achievement recompute/revocation (§12.3) | `achievement.user-progress`: `unlock`, `bulk-unlock`, `reset`, `list` | **GO, with the fallback already scoped** | `reset` covers admin-driven correction/revocation for simple stat-triggered achievements. There's no native streak/per-hand-condition trigger logic (only unlock/reset against a progress counter) — confirms the dev plan's existing fallback (Extend Event Handler computing event-log-derived counters) is required, not optional. No surprise here; existing §2.2a design already anticipated this correctly. |
| 8 | Challenge determinism/claim windows/catalog governance (§13.3) | `challenge.schedules` (`list`, `list-by-goal` — rotation is admin-configured, read via API), `challenge.progress` (`evaluate`, `list`/`list-my`, `list-past` — prior-rotation history, `claim-rewards*` — per-user and per-goal-code idempotent-shaped claim endpoints) | **CONDITIONAL GO** | Structurally a strong fit: rotation, progress-per-goal, past-rotation history, and reward claiming all exist natively, which is more complete than the dev plan's cautious framing suggested. Determinism of weekly *selection* (which 3 daily / 3 weekly missions from the catalog) is a config-time question (how schedules are authored), not an API-shape question — needs a hands-on config test, not just discovery, before dropping the `extend-challenge-suite` fork fallback. |
| 9 | Rank Suite fork vs MMR Calculator (§12.4–12.6) | `extend-rank-suite` (3-service suite: stats-listener/api/manager, Redis-backed) vs `extend-mmr-calculator` (single Service Extension) | **GO — reverses the dev-plan's stated preference order** | `extend-rank-suite`'s actual architecture is a **weekly-batch, 5-tier (Bronze–Diamond), 20-player synthetic-group** ranking system evaluated every Sunday — not a per-match system. Retrofitting it to §12.4's per-match pairwise Elo (K 40/24, immediate update, no tiers/groups) means gutting the rank-manager's weekly evaluation engine and the group-formation logic, far more than "replace its weekly-MMR logic" implied. `extend-mmr-calculator` is already invoked **per match result** (`POST /v1/mmr/process-match-result` with teams/players/winning team), already reads/writes MMR via AGS Statistics, and already implements an ELO variant with team-size and underdog adjustments — its invocation shape matches §12.4 far more closely; only the formula constants and floor/zero-sum rules need replacing. **Recommendation: fork `extend-mmr-calculator`, not `extend-rank-suite`.** Update §2.2a and E13.F1 in the dev plan to reflect this. |
| 10 | Leaderboard 12-week cycles, region boards, provisional labels (§12.8) | `leaderboard.leaderboards` (create/update/list/get, no visible cycle field) + `social.stat-cycles`/`user-stat-cycles` (statistic-level cycle lifecycle) | **CONDITIONAL GO** | No cycle/season concept lives in the `leaderboard` service itself — AGS leaderboards are just ranked views over a stat. Season/12-week cyclicality has to be modeled at the underlying `social` stat via `stat-cycles`, which the leaderboard then reads. Per-region + global boards are just separate leaderboard codes (one per region + one global), which `leaderboards.create`/`list` supports trivially. Provisional-rating labeling (10-match eligibility) is not an AGS concept at all — matches the dev plan's existing design of first-party Go computing eligibility/provisional labels before publishing to AGS leaderboards. No design change; confirms the existing plan. |
| 11 | Inbox claim idempotency, per-item Claim All, expiry (§10.9) | `chat.inbox`: `send-message`, `create-message`, `list-messages`, `list-stats`, `list-users`, `unsend-message`, categories — **no `claim` method anywhere in `chat.inbox`.** Reward fulfillment lives in a different service: `platform.rewards` (`create`/`match-event`/`export`/`import`) alongside `platform.fulfillments`. | **FALLBACK confirmed** | AGS's admin inbox is messaging/notification only — it has no claim, no per-item claim state, and no expiry field in the described surface. Reward *fulfillment* (if items/Jade are attached to a mail item) is a separate `platform` capability with its own semantics, not part of `chat.inbox`. This confirms the dev plan's already-scoped fallback — the Go claim layer (idempotent claim, per-item Claim All, expiry) is **required, not optional** — and clarifies that "mail" and "reward fulfillment" are two different AGS services that the claim layer has to bridge, which the current plan text doesn't call out explicitly. |
| 12 | Game Telemetry field controls + export (§15.3) | `game-telemetry.events.search` (admin query), `game-telemetry.namespaces.list` | **CONDITIONAL — confirms existing design, doesn't replace it** | The admin API surface only exposes event *querying*, not a field-level allowlist/redaction control — field control is a client/SDK-side concern (what the event payload contains when sent), not a server-enforced gate visible via `describe`. This is consistent with — and reinforces the necessity of — the dev plan's existing design (client-enforced field allowlist ahead of the schema registry, §2.2a). No native AGS enforcement was found, so that first-party control stays load-bearing. Warehouse export path not verified via CLI (likely a data-pipeline/Admin-Portal concern outside CLI scope) — follow-up if export mechanics matter before Beta. |
| 13 | Reporting category/evidence fit (§10.8) | `reporting.reports.create` body: `category`, `reason` (predefined lists), `extensionCategory`, `objectId`/`objectType` (UGC), `additionalInfo` (free-form JSON), `comment` (**max 280 chars**); `reporting.reasons`/`reasons.create-group` (configurable taxonomy); `reporting.extensions` (auto-moderation categories/actions) | **CONDITIONAL GO, one concrete copy/UI gap** | Category taxonomy is fully admin-configurable (reasons + groups), and `additionalInfo` is exactly the free-form slot the plan's Event Handler can use to auto-attach the Match-ID evidence bundle (match ID, event slice, versions) — good fit. **Gap: the native `comment` field caps at 280 characters; the spec (§10.8) calls for 500.** Either move the reporter's free text into `additionalInfo` (uncapped JSON) instead of `comment`, or the spec's copy needs to accept the AGS 280-char ceiling — a decision for the Product Owner, not an engineering workaround. |
| 14 | Deletion/export coverage vs §10.4 | `gdpr.user-deletion-requests` (`submit`/`submit-my`, `get`/`get-my`, `list-all`, `cancel-all`/`cancel-all-my`) and `gdpr.user-access-requests` (`submit`, `list`/`list-all`, `generate-download-url`, `cancel`) | **GO for AGS-held data** | Strong native match: submit/cancel/status for deletion (the `cancel-all` pair maps naturally onto the spec's 30-day cancellation window) and submit/list/download-URL for export requests. This covers AGS-held identity/profile data only — app-owned stores (rules-engine event log, Jade journal, T&S records) still need first-party purge/export jobs orchestrated against these AGS requests, exactly as §2.2a already scopes it (open item: build that orchestration as a Service Extension). No design change; confirms the plan. |

## Follow-ups this spike surfaced (not blockers, but should not be silently dropped)

1. Confirm IAM refresh-token expiry semantics (absolute vs. sliding/inactivity) with AccelByte docs or support — item 2.
2. Decide the month/year-DOB-vs-self-attestation trade-off for the age gate with the Product Owner — item 3 (this is a product decision, not an engineering one).
3. Hands-on config test of Challenge weekly-selection determinism before dropping the `extend-challenge-suite` fork fallback — item 8.
4. **Update dev-plan §2.2a and E13.F1**: swap the stated rank fork preference from `extend-rank-suite` to `extend-mmr-calculator` — item 9. This is a real correction to the existing plan, not just a spike note.
5. Decide the 280-char AGS report-comment cap vs. the spec's 500-char requirement — item 13.
6. Verify Game Telemetry warehouse export mechanics if export timing matters before Beta dashboards (E15) — item 12.
